Spring Boot 3 升级避坑指南：全局 Jackson 配置为何“背刺”了 Knife4j？

1. 雪花算法与 JavaScript 的“精度之痛”

在分布式系统中，雪花算法（Snowflake） 是生成分布式 ID 的首选。它生成的 19 位 Long 型 ID 在 Java 端运行完美，但到了前端却会引发一场灾难。

矛盾点： JavaScript 的 Number 类型遵循 IEEE 754 标准，其最大安全整数为 $2^{53} - 1$（即 9007199254740991，16 位）。当 19 位的 Long 型 ID 传给前端时，低三位会因精度丢失而自动变为 0，直接导致业务主键失效。

为了彻底解决这一问题，在 Spring Boot 2.x 时代采用了一个“一劳永逸”的方案：在 WebMvcConfiguration 中重写 extendMessageConverters，将自定义的 Jackson 转换器强行插入到 第 0 位（最高优先级），强制将所有 Long 类型序列化为 String。

这段代码在 Swagger 2 时代一直表现稳健，直到我们将项目推向了 Spring Boot 3。

异常现象观测：

  1. 接口响应异常：通过浏览器 F12 观察 /v3/api-docs 接口，发现返回的不是预期的 JSON 结构，而是一串极其冗长的 Base64 字符串。
  2. 前端解析失败：由于响应格式完全不符合 OpenAPI 规范，Knife4j 无法解析文档定义，最终导致页面渲染逻辑崩溃。
  3. 状态码困惑：即便强制刷新，接口有时返回 304，有时返回 200，但 Payload 始终是那串 Base64。

3. 深度溯源：谁动了我的 JSON？

为什么同样的全局配置，在 Spring Boot 2 中平安无事，在 Spring Boot 3 中就成了毒药？

原因一：Springdoc 的底层优化

在 Spring Boot 3 对应的 Springdoc 实现中，为了提升性能，接口定义（OpenAPI Spec）在某些场景下会直接以 byte[]（字节数组）的形式返回。

原因二：转换器优先级的“回旋镖”

我们在 extendMessageConverters 中将自定义 Jackson 转换器放到了 0 号位置。这意味着：任何类型的响应都会优先询问这个转换器：“你能处理吗？”

原因三：Jackson 的默认行为

当 byte[] 流经我们自定义的 MappingJackson2HttpMessageConverter 时，Jackson 发现自己可以处理字节数组，而它的默认处理逻辑就是：将字节数组序列化为 Base64 字符串。

结论： 我们的全局配置由于优先级过高，意外“拦截”了本该由 ByteArrayHttpMessageConverter 处理的文档流，强行把 JSON 字节流转码成了 Base64 文本。

4. 解决方案：优雅地“放行”

解决思路很简单：让自定义转换器学会“拒绝”它不该管的内容。 我们需要重写 canWrite 方法，将 byte[] 类型排除在外。

5. 经验总结：升级路上的三条铁律

  1. 慎用“绝对优先级”：通过 converters.add(0, ...) 增强功能时，务必考虑对非 JSON 资源（图片、二进制流、PDF、接口文档）的影响。
  2. Spring Boot 3 的透明变化：框架升级不仅是包名的重构（javax -> jakarta），底层数据流转逻辑（如 Springdoc 对 byte[] 的使用）的改变往往更隐蔽。
  3. 调试排查路径：当 UI 框架（如 Knife4j/Swagger）报错时，第一时间看 Network 原始响应。如果响应内容与 Content-Type 不符，通常是 HttpMessageConverter 链条发生了非预期的拦截。

作者后记：技术升级从不是简单的版本号更迭，而是对底层原理的又一次深度复习。希望这个“坑”能帮你少走弯路。
