从本地锁到分布式锁：Redis 分布式锁深度解析与 Redisson 实战

摘要：本文从超卖问题出发，系统梳理了本地锁到分布式锁的演进过程，深入剖析 Redis 分布式锁的实现原理、潜在问题及解决方案，并详细解读 Redisson 框架的可重入锁、看门狗、MultiLock 等核心机制。

一、什么时候需要锁，以及锁是干什么的？

1.1 超卖问题场景

在多线程环境中，如果多个线程同时访问共享资源（例如商品库存、外卖订单），会发生数据竞争，可能导致脏数据或系统问题，威胁程序的正常运行。

典型场景：在点评类项目中，100 个用户参与限时秒杀活动，每位用户限购 1 件商品，且商品数量只有 3 个。如果不对共享资源进行互斥访问，可能出现以下情况：

  • 线程 1、2、3 等多个线程同时进入抢购方法，每一个线程对应一个用户
  • 线程 1 查询用户已抢购数量，发现当前用户尚未抢购且商品库存还有 1 个，认为可以继续执行抢购流程
  • 线程 2 也执行查询，发现当前用户尚未抢购且商品库存还有 1 个，认为可以继续执行抢购流程
  • 线程 1 继续执行，将库存数量减少 1 个，然后返回成功
  • 线程 2 继续执行，将库存数量减少 1 个，然后返回成功
  • 此时就发生了超卖问题，导致商品被多卖了一份

1.2 锁的本质

为了解决这一情况，我们需要加锁，保证互斥性——即同一时刻只允许一个线程访问共享资源，其他线程需要等待当前线程释放后才能访问。这样可以避免数据竞争和脏数据问题，保证程序的正确性和稳定性。

二、本地锁方案：悲观锁 vs 乐观锁

2.1 悲观锁

核心思想：认为线程安全问题一定会发生，因此操作数据库之前都需要先获取锁，确保线程串行执行。

常见实现：synchronized、Lock（ReentrantLock）

性能建议：使用锁尽量锁代码块，不锁方法。锁的范围越大，性能就会越低。

2.2 乐观锁

核心思想：认为线程安全问题不一定发生，因此不加锁，只会在更新数据库时判断有没有其它线程对数据进行修改。

  • 如果没有修改则认为是安全的，直接更新数据
  • 如果修改了则说明不安全，抛异常或等待重试

常见实现：版本号法、CAS 操作

2.3 悲观锁与乐观锁对比

维度 悲观锁 乐观锁
性能 低（需先加锁再操作） 高（无需加锁）
冲突处理 阻塞其他线程 提交阶段检查冲突并重试
并发度 低（锁粒度大） 高
适用场景 写入操作多、冲突频繁 读取操作多、冲突较少

三、分布式锁的引入

3.1 本地锁的局限性

在单体项目中使用 synchronized 或 Lock 没问题，但在集群或分布式环境中就会出问题。

根本原因：synchronized 是本地锁，只能提供线程级别的同步。每个 JVM 中都有一把 synchronized 锁，不能跨 JVM 进行上锁。

当创建了两个节点（两个 JVM），synchronized 会失效！

3.2 分布式锁的应用场景

系统的订单服务部署了 3 份，都对外提供服务。用户下订单前需要检查库存，为了防止超卖，需要加锁以实现对检查库存操作的同步访问。由于订单服务位于不同的 JVM 进程中，本地锁无法正常工作，需要用到分布式锁——即使多个线程不在同一个 JVM 进程中，也能获取到同一把锁，实现共享资源的互斥访问。

3.3 分布式锁的核心特点

特性 说明
多线程可见 所有节点都能看到锁的状态
互斥 任何时刻只有一个节点能获得锁
高可用 网络分区或节点故障时仍能正常工作
容错性 持有锁的节点故障时，系统能自动释放锁
高性能 减少共享资源的访问等待时间和锁竞争开销
可重入性 已持有锁的节点可再次获取同一把锁，避免死锁
锁超时机制 避免无限期持有锁，确保锁的自动释放

核心原则：不论是本地锁还是分布式锁，核心都在于**”互斥”**。

4.2 问题分析：误释放锁

场景1：线程 1 获取锁后，由于业务阻塞，锁超时释放了。此时线程 2 趁虚而入拿到锁，然后线程 1 业务完成，把线程 2 刚刚获取的锁给释放了。接着线程 3 又趁虚而入，导致再次出现超卖问题！

五、Redis 分布式锁的优化演进

5.1 优化 1：线程标识防误释放

为分布式锁添加线程标识，在释放锁时判断当前锁是否是自己的：

  • 是自己的锁 → 直接释放
  • 不是自己的锁 → 不释放

六、生产级分布式锁的四大挑战

6.1 不可重入问题

不可重入：同一线程不能重复获取同一把锁。

场景：方法 A 调用方法 B，两者都需要获取同一把分布式锁。线程 1 进入方法 A 获取一次锁，进入方法 B 又获取一次锁，由于锁不可重入，导致死锁。

6.2 不可重试问题

获取锁只尝试一次就返回 false，没有重试机制。这会导致数据丢失——线程 1 要写入数据库，但锁被线程 2 占用，线程 1 直接结束而不重试。

6.3 超时释放问题

超时释放机制虽避免了死锁，但如果业务执行耗时过长，期间锁被释放，存在安全隐患。

困境：锁有效期过短 → 业务没执行完就被释放；锁有效期过长 → 容易出现死锁。

解决方案：设置较短有效期 + 心跳机制与自动续期（Watch Dog）

6.4 主从一致性问题

Redis 主从集群中，主从同步存在延迟：

  • 线程 1 在 Master 获取锁
  • Master 未同步给 Slave 就宕机
  • Slave 升级为新 Master，但没有线程 1 的锁信息
  • 线程 2 请求加锁，在新 Master 上加锁成功 → 两个线程同时持有锁

七、Redisson：生产级分布式锁框架

7.1 为什么选择 Redisson？

手动优化分布式锁可以，但没必要。Redisson 是一个成熟的 Redis 框架，功能包括：

  • 分布式锁和同步器
  • 分布式对象
  • 分布式集合
  • 分布式服务

Redisson 是使用 Redis 解决分布式问题的方案集合。

7.2 可重入锁（Reentrant Lock）

可重入锁：同一线程可多次获取同一把锁。如 Java 中的 synchronized 和 ReentrantLock。

实现核心：为每个锁关联可重入计数器和占有线程。当可重入计数器 > 0 时，判断占有线程和请求线程是否为同一个。

实现原理：利用 Hash 结构记录线程 ID 和重入次数

  • 获取锁时：判断锁中的线程 ID 是否一致，一致则重入次数 +1
  • 释放锁时：判断线程 ID，重入次数为 0 时真正释放，否则无视

7.5 主从一致性与 MultiLock

问题背景：Redis 主从架构是异步复制的，存在主从切换导致锁丢失的风险。

解决方案：MultiLock（联锁），基于 RedLock 算法的规范实现。

核心思路：不再只向一个 Master 请求锁，而是向多个相互独立的 Redis 节点请求锁。只有当”大多数”（N/2 + 1）或”全部”节点都加锁成功时，才认为最终加锁成功。

原理总结：

  1. 多节点锁定：向一组独立的 Redis 实例（通常 3 或 5 个）依次申请加锁
  2. 防脑裂/宕机：即使一个 Master 宕机且数据丢失，其他节点仍持有锁信息，新线程无法获得”大多数”节点同意
  3. 回滚机制：部分成功、部分失败时，自动释放已加锁成功的节点

应用建议：

场景 推荐方案
追求极致高可用与一致性 RedissonMultiLock
普通业务 标准 RLock + 主从架构（内网环境下主从丢失概率极低，业务层通常有补偿机制）
金融结算等严苛场景 必须使用 MultiLock

八、Redisson 分布式锁原理总结

特性 实现机制
可重入 Hash 结构记录线程 ID 和重入次数，同线程直接获取并计数
可重试 信号量 + Pub/Sub 实现等待、唤醒、失败重试，避免 CPU 空转
超时续约 Watch Dog 每隔 leaseTime/3（默认 10s）重置超时时间
主从一致性 MultiLock 多节点锁定，需大多数节点加锁成功

缺陷：运维成本高、实现复杂。

九、总结

从本地锁到分布式锁的演进，本质上是并发控制范围从单 JVM 扩展到多节点的过程。Redis 分布式锁通过 SETNX 实现基础互斥，通过 Lua 脚本保证原子性，但仍面临可重入、可重试、超时释放、主从一致性等挑战。

Redisson 框架提供了完整的生产级解决方案：

  • 可重入锁解决嵌套调用死锁问题
  • 看门狗机制自动续期避免业务中断
  • Pub/Sub + 信号量实现高效锁重试
  • MultiLock应对主从一致性挑战

在实际应用中，应根据业务场景选择合适的方案：普通业务使用标准 RLock 即可，金融级强一致性场景才需引入 MultiLock。
